# Member Value And Programming

## Value Architecture

Define value by participant role. A new member, experienced practitioner, expert, moderator, partner, customer, and employee can need different outcomes and carry different responsibilities.

| Layer | Question |
| --- | --- |
| Relevance | Why is this person eligible and why does the purpose matter now? |
| Orientation | What context, norms, identity, and role help them participate safely? |
| First value | What useful outcome can occur without requiring performative activity? |
| Repeated value | Which interaction, knowledge, relationship, practice, or product path recurs naturally? |
| Contribution | What can the participant create, answer, host, connect, review, or improve voluntarily? |
| Recognition | How is useful contribution credited without creating unhealthy status pressure? |
| Product path | How can community value support first or repeated product value without becoming disguised promotion? |
| Exit and return | How can a participant reduce notifications, leave, delete data, or return without penalty? |

Observers can receive value through trusted knowledge, examples, or belonging. Do not require everyone to post.

## Cold-Start Design

1. Choose one narrow participant job and smallest viable community unit.
2. Recruit or invite only relevant seed participants with permission and an honest promise.
3. Supply credible hosts, expertise, questions, examples, and response capacity.
4. Design one repeatable value-bearing interaction.
5. Resolve empty states, unanswered needs, newcomer uncertainty, and unsafe behavior manually within declared capacity.
6. Measure first and recurring value before scaling acquisition.
7. Expand roles, programs, segments, or platforms only when quality and operations remain controlled.

Do not simulate members, fake replies, prefill personal profiles, buy lists, or force customers or employees into membership. Label staff participation and sponsorship honestly.

## Onboarding Path

Design the minimum path from voluntary entry to first value. It may include purpose, consent, rules, notification choices, profile or context, role selection, introduction, matching, starter resource, first question, first response, first event, first contribution, or product handoff. Use only steps required by the member job and safety model.

Define recovery for incomplete profile, no match, no response, invalid invitation, access failure, language mismatch, notification overload, and changed intent. Joining, reading rules, or receiving a welcome message is not necessarily first value.

## Program Portfolio

Assign one primary job and value outcome to each program:

| Program family | Possible member value | Common failure |
| --- | --- | --- |
| Orientation | Understand norms, roles, resources, and first step | Information dump without first value |
| Questions and support | Resolve a specific need | Repeated unanswered asks or hidden support substitution |
| Peer exchange | Learn from comparable practice or experience | Generic networking and status competition |
| Expert session | Access credible judgment and context | Broadcast promotion without interaction |
| Practice or challenge | Apply a skill with feedback | Activity streak without durable value |
| Working group | Produce a useful shared outcome | Unclear authority or unpaid hidden labor |
| Knowledge contribution | Create reusable, attributable resources | Extraction without consent, credit, update, or removal |
| Product enablement | Reach first or repeated product value | Community used to mask product or support defects |
| Feedback and research | Influence decisions through bounded evidence | Biased sample presented as all customers |
| Advocacy | Share credible value voluntarily | Forced promotion or undisclosed incentive |

For every program define eligibility, cadence, host, contributor, preparation, interaction, output, reuse, product path, accessibility, moderation, capacity, cost, success, guardrails, review, and retirement. Retire programs that consume member and host capacity without decision-relevant value.

## Recognition And Incentives

Recognition can include attribution, access, learning, responsibility, status, compensation, or reciprocal support. Define why it helps the participant and community. Preserve opt-out, fair eligibility, transparent criteria, expiry, appeal, and role separation.

Do not reward message volume, conflict, recruiting pressure, always-on availability, or opaque popularity. Route material rewards, subsidies, credits, and incentive economics to `incentive-system-design`.

## Knowledge And Content Rights

For member-originated knowledge, record contributor, consent, quote and reuse rights, context, edit rights, attribution, translation, privacy, update owner, expiry, and removal. A conversation does not automatically authorize public content, model training, sales use, or product claims. Route editorial assets and distribution to `content-strategy`.
